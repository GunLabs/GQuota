私有仓库开源发布操作手册

从 private repository 走到 public repository 的完整流程：文档、License、secret scan、历史清理、GitHub 设置、远端复核。

给下次 AI 的操作摘要

目标：把一个私有 GitHub 仓库整理成可以公开的仓库。流程必须先清理当前树和历史，再公开。不要先点 public 再补救。

必须检查：当前源码 secret、git history secret、私人邮箱、本机路径、真实 fixture、构建产物、IDE 用户状态、.env、License、README 定位、贡献边界、Security policy、GitHub branch protection、secret scanning。

公开后必须重新 clone 远端验证。只看本地扫描不够。

适用范围

适用于个人或小团队把一个原本私有开发的仓库公开。典型场景：开发过程中有临时 commit、私人邮箱、本机路径、真实接口样本、旧 secret、草稿文档，公开前需要整理成可读、可维护、无明显泄漏的状态。

不适用于已经有外部用户依赖 commit hash、已有大量公开 fork、或必须保留完整审计历史的仓库。那种项目不应随意重写历史。

总流程

  1. 确认仓库状态：当前分支、远端、默认分支、GitHub 可见性。
  2. 补齐开源文档：README、LICENSE、CONTRIBUTING、SECURITY、TODO/roadmap。
  3. 清理当前源码树：secret、私人路径、真实样本、生成产物、IDE 用户状态。
  4. 扫描 git history：secret、私人邮箱、本机路径、真实凭据。
  5. 必要时重写历史：替换 secret、mailmap 邮箱、移除本机路径。
  6. 可选：公开前 squash 成一个 Initial public release root commit。
  7. 跑测试、构建、gitleaks、旧值搜索。
  8. 推送到远端。历史改写时必须使用 --force-with-lease。
  9. 配置 GitHub：描述、topics、secret scanning、push protection、vulnerability reporting、branch protection。
  10. 切换 public。
  11. 重新 clone 公开远端并复核。

开源文档准备

文件 必须说明
README.md 项目是什么、非官方声明、构建方式、隐私边界、风险、测试命令、License 链接。
LICENSE 明确许可证。常见个人开源项目可用 MIT，但要确认你确实想允许商业使用、修改、再分发。
CONTRIBUTING.md 接受什么贡献，不接受什么贡献，测试要求，fixture 脱敏规则，贡献默认 License。
SECURITY.md 如何报告安全问题，不要在 public issue 粘贴真实凭据，项目关注哪些安全范围。
TODO / roadmap 移除历史状态噪音，只保留真实开放项、外部依赖项、发布前阻塞项。
.gitignore 忽略 .env、构建目录、IDE 用户状态、系统文件和本地 agent 配置。

README 风险声明模板

This is an unofficial, local-first tool. It is not affiliated with the providers it queries.
It reads local credentials only to query the current user's own account data.
It does not upload tokens or quota responses to a project backend.
Provider endpoints, schemas, auth behavior, and terms may change at any time.

安全检查

顺序很重要：先扫描和清理，再公开。公开后再发现 secret，处理成本会高很多。

当前工作树扫描

rg -n --hidden --glob '!.git/**' --glob '!**/.build/**' \
  '(AKIA[0-9A-Z]{16}|sk-[A-Za-z0-9_-]{20,}|ghp_[A-Za-z0-9_]{20,}|github_pat_|xox[baprs]-|GOCSPX-|AIza[0-9A-Za-z_-]{20,})' .

rg -n --hidden --glob '!.git/**' --glob '!**/.build/**' \
  '(PRIVATE_EMAIL|OLD_LOCAL_PATH|/Users/LOCAL_USER|real-account-id|real-project-id)' .

gitleaks 扫描

gitleaks dir . --redact --no-banner
gitleaks detect --source . --redact --no-banner

Git history 搜索

git log --all --format='%an <%ae>' | sort -u
git grep -n -I -e 'OLD_SECRET_PATTERN' -e 'PRIVATE_EMAIL' -e 'OLD_LOCAL_PATH' $(git rev-list --all)

需要人工判断的命中

  • 真实 API key、OAuth client secret、GitHub token、cloud key：按泄漏处理，轮换或废弃。
  • 真实邮箱、账号 ID、project ID：能替换就替换，fixture 用 user@example.com、acct-FAKE123、proj-FAKE123。
  • 本机路径：替换为 仓库根目录 或 REDACTED_LOCAL_PATH。
  • 测试 fixture：保留结构，不保留身份和凭据字段。

历史清理

如果当前树已经干净，但 gitleaks detect 或 git grep $(git rev-list --all) 仍有命中，需要重写历史。

创建替换表和 mailmap

cat > /tmp/repo-filter-replace.txt <<'EOF'
literal:OLD_SECRET_VALUE==>REDACTED_SECRET
literal:OLD_LOCAL_PATH==>REDACTED_LOCAL_PATH
literal:PRIVATE_EMAIL==>PUBLIC_NOREPLY_EMAIL
EOF

cat > /tmp/repo-mailmap.txt <<'EOF'
PublicName <PUBLIC_NOREPLY_EMAIL> <PRIVATE_EMAIL>
EOF

执行 git-filter-repo

git status --short --branch
git filter-repo --force \
  --replace-text /tmp/repo-filter-replace.txt \
  --replace-message /tmp/repo-filter-replace.txt \
  --mailmap /tmp/repo-mailmap.txt

git-filter-repo 可能会移除 origin remote。执行后要恢复 remote，并重新跑历史扫描。

git remote add origin https://github.com/OWNER/REPO.git
gitleaks detect --source . --redact --no-banner
git log --all --format='%an <%ae>' | sort -u

不要把包含旧 secret 的 bundle、patch、报告文件长期留在 /tmp 或仓库目录。验证完成后清理临时文件。

公开前 squash 成一个初始提交

这是可选步骤。它的作用是让公开仓库从一个干净的 Initial public release 提交开始。安全清理完成后再做。

推荐方式：从干净 tree 生成 root commit

STAMP=$(date +%Y%m%d-%H%M%S)
REMOTE_MAIN=$(git ls-remote origin refs/heads/main | awk '{print $1}')
git branch "backup/pre-public-squash-$STAMP" "$REMOTE_MAIN"

BACKUP_REF="backup/pre-public-squash-$STAMP"
NEW_ROOT=$(printf 'Initial public release\n' | git commit-tree "$BACKUP_REF^{tree}")
git update-ref refs/heads/main "$NEW_ROOT"
git checkout -f main

验证 squash 结果

git rev-list --count main
git diff --stat "$BACKUP_REF" HEAD --
git ls-tree -r --name-only HEAD | rg '(^.*\.build/|xcuserdata|project\.xcworkspace|\.swiftpm)'

不要直接用：git switch --orphan 后接 git add -A。orphan 分支会保留旧索引状态，容易把构建产物、IDE 用户状态、测试输出一起带进初始提交。

GitHub 设置

仓库描述和 topics

gh repo edit OWNER/REPO \
  --description "Short project description."

gh api --method PUT repos/OWNER/REPO/topics --input - <<'JSON'
{"names":["topic-a","topic-b","topic-c"]}
JSON

开启安全功能

gh repo edit OWNER/REPO \
  --enable-secret-scanning \
  --enable-secret-scanning-push-protection \
  --delete-branch-on-merge \
  --enable-issues

gh api --method PUT repos/OWNER/REPO/private-vulnerability-reporting
gh api --method PUT repos/OWNER/REPO/vulnerability-alerts
gh api --method PUT repos/OWNER/REPO/automated-security-fixes

设置 main 分支保护

gh api --method PUT repos/OWNER/REPO/branches/main/protection --input - <<'JSON'
{
  "required_status_checks": null,
  "enforce_admins": false,
  "required_pull_request_reviews": null,
  "restrictions": null,
  "allow_force_pushes": false,
  "allow_deletions": false,
  "required_conversation_resolution": true
}
JSON

切换 public

gh repo edit OWNER/REPO \
  --visibility public \
  --accept-visibility-change-consequences

如果还需要 force push 更新重写后的历史，先临时把 allow_force_pushes 设为 true，推送后立刻改回 false。

收尾清理

  • 删除临时扫描报告、替换表、mailmap、clone 目录。
  • 删除本地备份分支，除非你明确需要短期保留。
  • 确认 /tmp/repo*、/tmp/*gitleaks* 没有残留敏感文件。
  • 确认本地 Git author 使用 public noreply email。
git config user.name
git config user.email
git branch --list 'backup/*'
ls /tmp | rg 'repo|gitleaks|filter|mailmap'

给 AI 的复用提示词

请阅读 docs/open-source-release-guide.html，并按这份手册把当前私有仓库准备成公开仓库。
要求：
1. 先确认仓库状态、远端 main hash、GitHub 可见性和分支保护。
2. 补齐 README、LICENSE、CONTRIBUTING、SECURITY、TODO/roadmap 和 .gitignore。
3. 扫描当前树和 git history 的 secret、私人邮箱、本机路径、真实 fixture。
4. 如果历史有泄漏，用 git-filter-repo 清理，并重新验证。
5. 如需 squash，只能从已验证干净的 tree 用 git commit-tree 生成 root commit，不要 orphan + git add -A。
6. 所有历史重写推送必须使用 --force-with-lease。
7. 公开前后都要跑测试、gitleaks、旧值搜索。
8. 公开后开启 GitHub secret scanning、push protection、Dependabot security updates、private vulnerability reporting 和 branch protection。
9. 最后重新 clone 远端仓库验证，并清理所有临时文件。
